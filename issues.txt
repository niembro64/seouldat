Seouldat PeerJS / RTS-style lobby migration audit
==================================================

Reference project audited: /Users/ericniemeyer/code/rts-prototype
Current project audited: /Users/ericniemeyer/code/seouldat

Current state
-------------
- seouldat is a small Phaser 3.15 game served by Express from server/public.
- Networking is Socket.IO 2.1.1:
  - server/public/js/game.js connects with io(), sends playerInput, and renders server events.
  - server/authoritative_server/js/game.js runs a Phaser.HEADLESS authority in JSDOM and emits full playerUpdates every tick.
  - server/index.js boots Express, Socket.IO, JSDOM, canvas/datauri support, then listens on port 8082.
- Socket IDs are currently the player IDs. That couples identity, roster, reconnect behavior, and sprite ownership to Socket.IO.
- There is no lobby, room code, host/client role, player assignment, heartbeat, game-start handoff, or PeerJS signaling path yet.

rts-prototype pattern to adapt
------------------------------
- PeerJS transport lives in src/game/network/NetworkManager.ts.
- Room codes are short human codes mapped to PeerJS IDs by NetworkRoomCode.ts.
- Lobby state is managed as a roster by NetworkLobbyRoster.ts.
- Host and client setup are explicit:
  - hostGame() creates a PeerJS host with a room-derived peer ID.
  - joinGame(code) creates a random client peer ID and connects to the host peer ID.
  - incoming clients receive playerAssignment, playerJoined backfill, lobbySettings, and later gameStart.
- A heartbeat layer closes silent/disconnected peers while the lobby is open.
- The host owns lobby settings and broadcasts the whole settings object to clients.
- startGame() builds a battle handoff containing gameId, roomCode, playerIds, roster, and settings.
- The authoritative game server and render client talk through a GameConnection-style abstraction:
  - LocalGameConnection for the host/local client.
  - RemoteGameConnection for remote clients.
- Commands flow client -> host, snapshots flow host -> clients. The network layer has separate command and snapshot responsibilities.

Migration issues / work items
-----------------------------

[ ] 1. Decide the runtime shape before writing PeerJS code.
    Recommendation: make the browser host the authoritative server, like rts-prototype does, and retire the Node/JSDOM authoritative Phaser process after parity.
    Why: PeerJS is meant to connect browsers through WebRTC. Keeping the current Node JSDOM server as the authority would preserve the old central-server model and would not match the "other games" lobby/host flow.
    Acceptance:
    - Express only serves static files during/after migration.
    - The host browser owns simulation and sends snapshots.
    - Remote browsers only send input and render host state.

[ ] 2. Add PeerJS dependency and remove Socket.IO dependencies after the replacement is live.
    Current package.json dependencies to replace later: socket.io, jsdom, canvas, datauri.
    Options:
    - Minimal path: keep static script tags and load PeerJS from npm/CDN.
    - RTS-like path: introduce a Vite/module layout and import peerjs from package.json.
    Acceptance:
    - package.json includes peerjs.
    - /socket.io/socket.io.js is removed from server/public/index.html.
    - server/index.js no longer constructs SocketIO or JSDOM once PeerJS authority is working.

[ ] 3. Define a small seouldat network protocol before touching the game loop.
    Start with these message types, adapted from rts-prototype:
    - heartbeat: { type, gameId, playerId, playerInfo?, players? }
    - playerAssignment: { type, gameId, playerId }
    - playerJoined: { type, gameId, playerId, playerName }
    - playerLeft: { type, gameId, playerId }
    - lobbySettings: { type, gameId, settings }
    - gameStart: { type, gameId, playerIds, handoff, assignedPlayerId }
    - playerInput: { type, gameId, input }
    - state: { type, gameId, players, scores, star }
    Acceptance:
    - All network messages carry gameId or are explicitly setup-only.
    - Player identity uses stable numeric player IDs, not socket/peer connection IDs.
    - Host is always player 1.

[ ] 4. Add room-code utilities.
    Mirror rts-prototype/src/game/network/NetworkRoomCode.ts:
    - generate a 4-character human code with unambiguous characters.
    - normalize user-entered codes to uppercase.
    - convert room code to a PeerJS ID, for example sd-ABCD.
    Acceptance:
    - Host displays one room code.
    - Client can join with lowercase or whitespace-padded input.
    - Peer ID prefix is project-specific and will not collide with rts-prototype's ba-* IDs.

[ ] 5. Build a vanilla NetworkManager for seouldat.
    Reference: rts-prototype/src/game/network/NetworkManager.ts.
    Required behavior:
    - hostGame(): create Peer(roomCodeToGameId(code)), seed host roster, handle unavailable-id by retrying a new code.
    - joinGame(code): create random client Peer ID, connect to host, handle peer-unavailable and timeout errors.
    - handleIncomingConnection(): assign the next numeric player ID, reject joins after start, enforce max players.
    - setupConnectionHandlers(): handle data, close, error, and dataChannel monitoring.
    - safeSend(): central send wrapper that checks conn.open and catches send errors.
    - disconnect(): close connections, destroy peer, clear callbacks and timers.
    Acceptance:
    - Two browser tabs can host/join a lobby without starting the game.
    - Connection failures surface as lobby errors.
    - Leaving the lobby tears down PeerJS cleanly enough to host/join again without refresh.

[ ] 6. Add heartbeat and silent-disconnect cleanup.
    Reference: rts-prototype/src/game/network/NetworkHeartbeatTracker.ts.
    Required behavior:
    - Send heartbeat every ~2 seconds while in the lobby.
    - Mark a peer alive on any inbound message, not only heartbeat.
    - Close peers that have not sent anything for ~30 seconds before game start.
    Acceptance:
    - Closing a joined tab removes that player from the host and remaining clients.
    - Frozen or network-disconnected clients eventually leave the roster even if PeerJS close does not fire.

[ ] 7. Add lobby roster management.
    Reference: rts-prototype/src/game/network/NetworkLobbyRoster.ts.
    Required behavior:
    - Store playerId, name, isHost, and optional info fields.
    - Host seeds player 1.
    - Host backfills the full roster to a newly connected player.
    - Host broadcasts playerJoined/playerLeft/playerInfoUpdate to all clients.
    Acceptance:
    - All clients show the same lobby player list.
    - Local player row is identifiable.
    - Host badge and basic player names are present. IP/location/timezone can be deferred if not useful for this game.

[ ] 13. Create a remote client connection path.

    Suggested cadence: 20-30 Hz initially, then tune.
    Add basic backpressure checks with conn.dataChannel.bufferedAmount before sending large or frequent states.
    Acceptance:
    - Clients render smooth enough for the current game.
    - A slow client does not crash the host send loop.
    - A periodic full state is enough to recover from missed messages.
